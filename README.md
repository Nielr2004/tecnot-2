# Natural Language → SQL Engine

A Streamlit application that lets you query an e-commerce SQLite database using plain English.
The app sends your question to a Hugging Face instruct-tuned LLM, receives a SQL `SELECT`
statement, validates it, executes it, and shows you the results — all in one page.

---

## Quick Start

### 1. Clone / copy the project

```bash
cd tecnots-Q2
```

### 2. Create and activate a virtual environment

```bash
# Windows
python -m venv .venv
.venv\Scripts\activate

# macOS / Linux
python -m venv .venv
source .venv/bin/activate
```

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

### 4. Configure your Hugging Face token

```bash
cp .env.example .env
# Open .env and set HF_TOKEN=hf_your_token_here
```

Generate a token at <https://huggingface.co/settings/tokens>.
The token needs the **"Make calls to Inference Providers"** permission.

### 5. Seed the database (**run once, not on every app start**)

```bash
python seed.py
```

This creates `ecommerce.db` in the project root with 60 customers, 60 products,
80 orders, and 150+ order-items generated by Faker.

### 6. Run the app

```bash
streamlit run app.py
```

Open <http://localhost:8501> in your browser.

## Tech Stack & Rationale

| Layer | Choice | Why |
|---|---|---|
| **UI** | Streamlit | Zero-config, Python-native, great for rapid data tools |
| **Database** | SQLite via `sqlite3` + pandas | File-based, zero setup; `read_sql_query` gives a DataFrame directly |
| **AI** | HF `InferenceClient` + `Qwen/Qwen2.5-7B-Instruct` | Free-tier capable, instruction-following, publicly accessible on HF Hub |
| **Env secrets** | `python-dotenv` | Keeps tokens out of source; Streamlit Secrets also supported |
| **Fake data** | Faker | Realistic names/emails/dates with zero manual CSV work |

## Database Schema & Rationale

```
customers  ──< orders  ──< order_items >── products
```

| Table | Purpose |
|---|---|
| `customers` | Identity + location; `signup_date` enables cohort/trend queries |
| `products` | Catalogue with `category` + `price` + stock for inventory queries |
| `orders` | Transaction header; `status` enables funnel/cancellation analytics |
| `order_items` | Line-item detail; `unit_price` captures price-at-time-of-purchase (can differ from current `products.price`) |

The schema deliberately uses common SQL patterns — JOINs, GROUP BY, aggregations, date filters —
so that a wide variety of natural-language questions map cleanly to valid SQL.

## Assumptions & Trade-offs

### Open models vs. closed-model APIs

Closed frontier models (GPT-4o, Claude) follow system-prompt constraints reliably.
Open instruct models (Qwen, LLaMA) are "chattier" — they may:

- Wrap the SQL in markdown code fences (` ```sql ``` `)
- Prefix with "Sure! Here's the SQL:" preamble
- Occasionally ignore the "no DML" instruction

**Mitigations implemented:**

1. `_extract_sql()` in `nl_to_sql.py` strips fences and preamble with regex before treating
   the response as a query.
2. A second retry with a *stricter* system prompt fires if the first response can't be parsed.
3. `is_safe_query()` in `db.py` is a hard server-side guard — regex/keyword-checks that the
   query starts with `SELECT` and contains no blocked DML/DDL keywords, *regardless* of what
   the model returns.
4. The SQLite connection is opened in **read-only URI mode** (`?mode=ro`) as a final defence.

## Bonus Features Implemented

- **Editable SQL**: The generated SQL is rendered in a `st.text_area` so you can tweak it
  before running, with a dedicated "▶ Run SQL" button.
- **CSV Export**: A `st.download_button` on every result set uses `df.to_csv()`.
- **Schema Explorer**: The sidebar "📋 Schema Explorer" expander shows the full schema text
  (the same text sent to the model).
- **Query History**: Every query is stored in `st.session_state["history"]`; the sidebar lists
  them with a "↩ Reload" button that restores the NL input, SQL, and results.
- **Example queries**: A quick-start expander with five clickable example questions.

## Query Speed & Caching (Design Note)

### Current behaviour

Every "Run Query" press calls the HF Inference API (network-bound, ~2–5 s) and then
executes the SQL locally (sub-millisecond for this dataset size).

### A caching strategy worth implementing

**Key idea:** normalise the NL input (lowercase, strip punctuation, collapse whitespace) and
use it as a cache key mapping to `(generated_sql, result_df)`. A plain `dict` in
`st.session_state` already gives per-session deduplication — identical questions don't hit
the API twice.

**TTL invalidation:** wrap the cache entries with a `cached_at` timestamp; discard entries
older than N minutes. This handles data freshness: if rows were added to the DB the cached
result becomes stale, and a fresh query reflects the update.

**Where a semantic/LLM cache helps:** two *differently-phrased* questions with the same
intent ("top spenders" vs "highest revenue customers") miss a plain key-value cache but
would hit a semantic cache (embed both queries, compare cosine similarity, return cached
result if similarity > threshold). The trade-off is added latency for the embedding step and
the risk of false-positive cache hits when questions are superficially similar but logically
different. For a small demo tool a plain normalised-string cache is preferable; a semantic
cache becomes worthwhile at scale when LLM API costs or rate limits are significant.
